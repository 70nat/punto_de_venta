Seguimiento y Control del Proyecto con Git y GitHub
Este documento registra los cambios más relevantes del proyecto y muestra cómo cada commit se relaciona con actividades de la metodología híbrida Cascada + Scrum aplicada al desarrollo del Punto de Venta en C++.
El repositorio se utilizó como herramienta principal para monitorear avances, documentar decisiones y evidenciar el progreso del proyecto.
1. Relación entre commits y fases de la metodología híbrida
Fase 1: Análisis (Cascada)
Commit: Commit inicial en main
Descripción: Se establece la estructura base del repositorio y se define el alcance general del proyecto.
Relación metodológica: Actividad típica de la fase de análisis, donde se prepara el entorno y se documentan los primeros requisitos.
Fase 2: Diseño (Cascada)
Commit: Archivo base para conflicto / Base para conflicto real
Descripción: Se crean archivos iniciales probando estructura y funcionamiento.
Relación metodológica: Corresponde a la etapa donde se definen componentes y comportamiento del sistema, previo a programar en serio.
2. Implementación Iterativa (Scrum)
Sprint 1: Módulo inicial y experimentación con ramas
Commits:
Cambio desde rama A
Cambio desde rama B
Conflicto resuelto
Descripción: Se crean ramas paralelas para simular desarrollo simultáneo y se resuelve un conflicto de merge.
Relación con Scrum: Trabajo iterativo, manejo de integración continua y validación de incrementos.
Sprint 2: Documentación técnica
Commit: Agrego documentación metodológica
Descripción: Se integra la documentación relacionada con los pasos del proyecto y su estructura interna.
Relación con Scrum: Representa un “incremento” documental del sprint.
Sprint 3: Metodología híbrida
Commits:
Agrego documentación de metodología híbrida (Cascada + Scrum)
Pull Request + Merge hacia main
Descripción: Se genera una rama específica para mantener la documentación organizada y se integra mediante un Pull Request.
Relación con Scrum: Entregable del Sprint 3, revisión, aprobación y consolidación del trabajo.
3. Control del proyecto mediante GitHub
Git y GitHub permitieron:
✔ Seguimiento de avances
Cada commit actuó como marcador del progreso: decisiones, pruebas, correcciones y documentación.
✔ Control de versiones
Permite volver a cualquier punto del proyecto si es necesario.
✔ Evidencia de metodología
El orden de commits refleja fases claras:
Cascada al inicio
Scrum en iteraciones posteriores
Integración continua mediante PRs
✔ Organización con ramas
Separar código, documentación y experimentos evitó caos en el repositorio.
4. Conclusión
El repositorio sirvió no solo para almacenar código, sino como herramienta de seguimiento y control del proyecto.
Cada commit evidenció un paso dentro del modelo híbrido, permitiendo visualizar cómo evolucionó el sistema y asegurando orden, trazabilidad y claridad metodológica.
